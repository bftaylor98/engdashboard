1. PROJECT OVERVIEW
================================================================================

This is an internal browser-based dashboard application for tracking engineering
work orders, schedules, and project status in a manufacturing/machining shop
environment. It is used to manage engineering programming tasks (CNC/machining
work orders) across multiple customers and assignees.

The app is called "Engineering Schedule" and serves as a centralized hub for:
  - Viewing and managing engineering work orders
  - Tracking part programming assignments (who is programming what)
  - Monitoring material status (ordered, arrived, not ordered)
  - Linking out to the EST/Adion Systems ERP (est.adionsystems.com/procnc)
  - Kanban board for visual work order management
  - Analytics with workload, assignee breakdown, and construction metrics
  - Revision alert tracking (parts needing rev updates)
  - Excel import/export for bulk data management
  - Calendar view of work orders by due date
  - Real-time updates across all connected clients via SSE

The system is designed for a small team of engineers/programmers:
  - Rob, Damien, Thad, Alex, Phillip, Brad, Mike
  - "Prog. Completed", "Prog. Que", "Review", "Eng Comp" are special
    pseudo-assignees indicating various completion/queue states

Customers include: EST, SSP, PFI, ICI, AICHI, MERITOR, AAM (and others can
be added via import)

4. DATABASE SCHEMA
================================================================================

DATABASE: SQLite via better-sqlite3 (file: database/engineering_schedule.db)
Auto-created on first server start if it doesn't exist.
WAL journal mode enabled for better concurrent read performance.
Foreign keys enabled.
Default users are seeded if the users table is empty.

TABLE: engineering_work_orders
--------------------------------------------------------------
  id                    TEXT PRIMARY KEY
  wo_number             TEXT NOT NULL
  priority              INTEGER NOT NULL DEFAULT 0 CHECK(0-11)
  is_hot_job            INTEGER NOT NULL DEFAULT 0
  due_date              TEXT (nullable)
  part_number           TEXT NOT NULL
  rev_alert             TEXT (nullable)
  part_name             TEXT NOT NULL
  project               TEXT (nullable)
  qn                    TEXT (nullable)
  customer              TEXT NOT NULL
  est_programming_hours REAL (nullable)
  est_engineering_hours  REAL (nullable)
  price                 REAL (nullable)
  material_status       TEXT DEFAULT 'not-ordered'
  notes                 TEXT (nullable)
  comments              TEXT DEFAULT '[]'   (JSON stringified array)
  current_box           TEXT (nullable)     (assignee/box name)
  machine_scheduled     TEXT (nullable)
  current_status        TEXT NOT NULL DEFAULT 'assigned'
  metadata              TEXT DEFAULT '{}'   (JSON – extra import columns)
  version               INTEGER NOT NULL DEFAULT 1  (optimistic concurrency)
  created_at            TEXT NOT NULL       (ISO 8601)
  updated_at            TEXT NOT NULL       (ISO 8601)

  Indexes: wo_number, priority, current_box, customer, current_status, due_date

TABLE: revision_alerts
--------------------------------------------------------------
  id                INTEGER PRIMARY KEY AUTOINCREMENT
  part_number       TEXT NOT NULL
  part_name         TEXT (nullable)
  revision_date     TEXT (nullable)
  linked_wo_id      TEXT (nullable)
  created_at        TEXT NOT NULL
  updated_at        TEXT NOT NULL

  Index: part_number

TABLE: construction_metrics
--------------------------------------------------------------
  id                  INTEGER PRIMARY KEY AUTOINCREMENT
  box_name            TEXT NOT NULL
  percentage_of_jobs  REAL (nullable)
  jobs_to_go          INTEGER (nullable)
  jobs_scheduled      INTEGER (nullable)
  total_jobs          INTEGER (nullable)
  assignee_counts     TEXT DEFAULT '{}'  (JSON: { "Rob": 5, "Thad": 3, ... })
  snapshot_date       TEXT NOT NULL
  created_at          TEXT NOT NULL

TABLE: import_history
--------------------------------------------------------------
  id              INTEGER PRIMARY KEY AUTOINCREMENT
  filename        TEXT NOT NULL
  rows_imported   INTEGER
  rows_skipped    INTEGER
  rows_errored    INTEGER
  import_report   TEXT           (JSON – full import report)
  imported_at     TEXT NOT NULL

TABLE: users
--------------------------------------------------------------
  id              TEXT PRIMARY KEY       (e.g. "user-rob")
  username        TEXT NOT NULL UNIQUE   (e.g. "rob")
  display_name    TEXT NOT NULL          (e.g. "Rob")
  password_hash   TEXT NOT NULL          (bcrypt hash)
  created_at      TEXT NOT NULL

  Default seeded users (password "changeme"):
    rob, damien, thad, alex, phillip, brad, mike, admin

TABLE: sessions
--------------------------------------------------------------
  token           TEXT PRIMARY KEY       (64 hex chars, crypto.randomBytes(48))
  user_id         TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE
  expires_at      TEXT NOT NULL          (7 days from creation)
  created_at      TEXT NOT NULL

  Indexes: user_id, expires_at

5. API ENDPOINTS
================================================================================

Base URL: http://localhost:3001/api
(Proxied from Vite dev server at localhost:5173 via /api prefix)

All endpoints except /api/auth/login and /api/health require a valid session
token via "Authorization: Bearer <token>" header. The /api/events and
/api/export endpoints also accept ?token=<token> for browser-native requests
(EventSource, direct download links).

AUTHENTICATION (/api/auth) – PUBLIC:
  POST /api/auth/login    -> { success, data: { token, user: { id, username,
                             displayName } } }
                             Body: { username, password }
                             Creates session token (7-day expiry)

AUTHENTICATION (/api/auth) – REQUIRES AUTH:
  GET  /api/auth/me       -> { success, data: { id, username, displayName } }
  POST /api/auth/logout   -> { success }
                             Deletes the session token

HEALTH – PUBLIC:
  GET  /api/health        -> { status, timestamp, workOrderCount }

WORK ORDERS (/api/work-orders) – REQUIRES AUTH:
  GET    /                -> Paginated list with filters:
                             ?search=, ?priority=, ?customer=, ?currentBox=,
                             ?currentStatus=, ?materialStatus=, ?isHotJob=,
                             ?sortBy=, ?sortOrder=, ?page=, ?limit=
                             Response: { success, data: [...], pagination: {
                               page, limit, total, totalPages } }
  GET    /:id             -> { success, data: WorkOrder }
  POST   /                -> Create new work order
                             Required: woNumber, partNumber, partName, customer
                             Returns 201 + broadcasts work-order:created SSE
  PATCH  /:id             -> Safe partial update with merge + optimistic
                             concurrency (version check). Only provided fields
                             are overwritten. Increments version.
                             Broadcasts work-order:updated SSE
  DELETE /:id             -> Delete work order
                             Broadcasts work-order:deleted SSE

  Sort fields: dueDate, priority, woNumber, customer, currentBox, createdAt,
               partNumber, partName, project (also accepts snake_case variants)
  Special: priority=0 always sorts to bottom

  DB-to-API field mapping:
    wo_number             -> woNumber
    is_hot_job            -> isHotJob (boolean)
    part_number           -> partNumber
    rev_alert             -> revAlert
    part_name             -> partName
    due_date              -> dueDate
    est_programming_hours -> estProgrammingHours
    est_engineering_hours -> estEngineeringHours
    material_status       -> materialStatus
    current_box           -> currentBox
    machine_scheduled     -> machineScheduled
    current_status        -> currentStatus
    metadata              -> metadata (parsed JSON object)
    comments              -> comments (parsed JSON array)
    created_at            -> createdAt
    updated_at            -> updatedAt

STATS (/api/stats) – REQUIRES AUTH:
  GET /api/stats/dashboard   -> { success, data: { total, overdue, dueThisWeek,
                                 hotJobs, completed, inProgress, assigned,
                                 totalProgrammingHours, totalEngineeringHours,
                                 materialNotOrdered } }
                                 (All computed live from database)
  GET /api/stats/workload    -> { success, data: [{ assignee, jobCount,
                                 programmingHours, engineeringHours,
                                 overdueCount }] }
  GET /api/stats/assignees   -> { success, data: [{ assignee, total,
                                 assigned, in-progress, ... }] }

REVISION ALERTS (/api/revision-alerts) – REQUIRES AUTH:
  GET /                      -> { success, data: [{ id, partNumber, partName,
                                 revisionDate, linkedWorkOrders: [{ id,
                                 woNumber, partName }], createdAt, updatedAt }] }

CONSTRUCTION METRICS (/api/construction-metrics) – REQUIRES AUTH:
  GET /                      -> { success, data: [{ id, boxName,
                                 percentageOfJobs, jobsToGo, jobsScheduled,
                                 totalJobs, assigneeCounts, snapshotDate,
                                 createdAt }] }

IMPORT (/api/import) – REQUIRES AUTH:
  POST /api/import/upload    -> Upload Excel file (multer: field "file")
                                Returns preview with sheets, headers, mappings,
                                sample rows, and filePath for confirm step
  POST /api/import/confirm   -> Confirm import

MACHINES (/api/machines) – REQUIRES AUTH:
  GET /api/machines         -> { success, data: { "VMX 84-1": [...], "VMX 64-1": [...], "VMX 64-2": [...] } }

PROSHOP (/api/proshop) – See PROSHOP_QUERIES_BRAIN.txt for full list. Includes:
  GET /tooling-expenses, /material-status, /open-pos, /ncrs/recent, /ncrs/last24h, /ncrs/by-assignee;
  GET /cost-analysis?woNumber=... (material cost + estimated minutes); POST /import-work-orders.

SSE (/api/events) – AUTH VIA QUERY STRING:
  GET /api/events?token=     -> Server-Sent Events stream

                                work-order:updated, work-order:deleted,

6. AUTHENTICATION SYSTEM
================================================================================

OVERVIEW:
  The app uses session-token authentication. No JWT library is used; tokens are
  random 48-byte hex strings stored in the sessions table. Passwords are hashed
  with bcryptjs (cost factor 10).

LOGIN FLOW:
  1. User enters username + password on the Login page
  2. Frontend POSTs to /api/auth/login
  3. Backend validates credentials against users table (bcrypt.compareSync)
  4. On success, generates 96-char hex token (crypto.randomBytes(48))
  5. Stores token in sessions table with 7-day expiry
  6. Returns { token, user } to frontend
  7. Frontend stores token in localStorage (key: "auth_token")
  8. AuthContext sets user state; App renders protected routes

TOKEN VALIDATION:
  - requireAuth middleware checks Authorization: Bearer <token> header
  - Falls back to ?token= query string for export downloads and SSE
  - Validates token exists in sessions table and hasn't expired
  - Expired tokens are deleted on access
  - Attaches { id, username, displayName } to req.user

LOGOUT FLOW:
  1. Frontend calls POST /api/auth/logout with Bearer token
  2. Backend deletes session from sessions table
  3. Frontend clears localStorage and resets AuthContext state

AUTO-LOGOUT:
  - api.ts fetchJSON intercepts 401 responses
  - Clears localStorage token and reloads the page (shows login)

FRONTEND AUTH STATE (AuthContext.tsx):
  - AuthProvider wraps the entire app in App.tsx
  - On mount, if a token exists in localStorage, validates it via GET /me
  - Provides: user, token, loading, login(), logout()
  - useAuth() hook for consuming components
  - While loading=true, a spinner is shown; if no user, Login page is shown

DEFAULT CREDENTIALS:
  - Seeded users: rob, damien, thad, alex, phillip, brad, mike, admin
  - Default password for all: "changeme"

7. REAL-TIME UPDATES (SSE)
================================================================================

OVERVIEW:
  Server-Sent Events provide one-way real-time updates from server to all
  connected clients. When any client creates, updates, or deletes a work order
  (or completes an import), all other connected clients receive the event and
  automatically refresh their data.

BACKEND ARCHITECTURE:
  - server/lib/eventBus.js: Singleton Node.js EventEmitter (max 100 listeners)
  - server/routes/events.js: SSE endpoint at GET /api/events?token=
    * Authenticates via query string token (EventSource can't set headers)
    * Sets SSE headers (text/event-stream, no-cache, keep-alive)
    * Sends initial "connected" event
    * Sends :heartbeat comment every 30 seconds
    * Listens on eventBus for 4 event types and forwards to client
    * Properly cleans up named listeners on client disconnect
  - workOrders.js: emits work-order:created, work-order:updated,
    work-order:deleted after successful mutations
  - import.js: emits import:completed after successful import

FRONTEND ARCHITECTURE:
  - src/hooks/useSSE.ts: Custom hook that:
    * Opens EventSource with token from AuthContext
    * Listens for all SSE event types
    * Calls onEvent callback when events arrive
    * Auto-reconnects after 3-second delay on error
    * Cleans up on unmount or token change
  - All data pages (Dashboard, Schedule, Kanban, Analytics, Calendar,
    RevisionAlerts) use useSSE to trigger data refetch on relevant events

EVENT TYPES:
  - work-order:created  -> payload: full WorkOrder object
  - work-order:updated  -> payload: full WorkOrder object
  - work-order:deleted  -> payload: { id }
  - import:completed    -> payload: import report
  - connected           -> payload: {} (initial connection confirmation)

11. DATA FLOW & STATE MANAGEMENT
================================================================================

State Management: React useState + useEffect + useCallback (no external state
library). Authentication state managed via React Context (AuthContext).

Authentication Flow:
  1. App mounts -> AuthProvider checks localStorage for token
  2. If token exists, calls GET /api/auth/me to validate
  3. If valid -> sets user, renders protected routes
  4. If invalid -> clears token, shows Login page
  5. All API calls include Bearer token via api.ts authHeaders()
  6. 401 responses trigger auto-logout (clear token + page reload)

Work Order Data Flow:
  1. Page component mounts (e.g. Schedule)
  2. useEffect triggers fetch via api.ts (includes auth headers)
  3. Express route validates token via requireAuth middleware
  4. Route queries SQLite, transforms snake_case -> camelCase
  5. Response sent with data + pagination metadata
  6. Component stores in local state, renders table/cards/etc.
  7. Mutations (create/update/delete) send POST/PATCH/DELETE to API
  8. API processes mutation, emits event to eventBus
  9. All SSE-connected clients receive the event
  10. useSSE hook triggers data refetch in each connected page

Real-Time Update Flow:
  1. Client A updates a work order (PATCH /api/work-orders/:id)
  2. Server saves to DB, emits 'work-order:updated' on eventBus
  3. SSE endpoint forwards event to all connected EventSource clients
  4. Client B's useSSE hook fires, calling the loadData/fetchData callback
  5. Client B re-fetches data from API and re-renders with latest state

Optimistic Concurrency:
  - Each work order has a `version` field (starts at 1)
  - PATCH endpoint checks if submitted version matches DB version
  - Returns 409 Conflict if versions don't match
  - Version is incremented on every successful update

13. IMPORT/EXPORT SYSTEM
================================================================================

IMPORT:
  - Accepts .xlsx, .xlsm, .xls, .csv files (max 50MB)
  - Two-step process: upload (preview) -> confirm (commit)
  - Auto-detects header row (scans first 5 rows, scores by known column names)
  - Fuzzy column mapping via COLUMN_MAP dictionary (e.g. "WO#" -> wo_number,
    "Part Number" -> part_number, "Current Box" -> current_box)
  - Imports 3 sheets by name detection:
    * Sheet containing "schedule" -> engineering_work_orders
    * Sheet containing "revision" -> revision_alerts
    * Sheet containing "construction" -> construction_metrics
  - Replace mode: clears all existing data before import
  - Append mode: adds to existing data
  - Material status normalization: "yes"/"received" -> arrived, etc.
  - Saves import history with report (rows imported/skipped/errored)
  - Cleans up uploaded file after import
  - Broadcasts import:completed SSE event

EXPORT:
  - XLSX export: 3-sheet workbook (Schedule, Needed Revisions, Construction)
  - CSV export: schedule data only
  - Timestamped filenames: Engineering_Schedule_YYYY-MM-DDTHH-MM-SS.*
  - Auth via query string token (for direct <a href> download links)

  non-API routes (catch-all route sends dist/index.html).

18. KEY DESIGN DECISIONS
================================================================================

1. BETTER-SQLITE3 OVER SQLITE3: Synchronous driver chosen for simpler code
   (no callback/promise overhead). WAL mode enables concurrent reads while
   one writer operates.

2. SESSION TOKENS OVER JWT: Simple random hex tokens stored in DB. Easy to
   invalidate (just delete the row). No JWT secret management needed. 7-day
   expiry provides reasonable session duration.

3. SSE OVER WEBSOCKETS: Server-Sent Events chosen for simplicity – one-way
   server-to-client push is sufficient for data refresh notifications. No
   WebSocket library needed; uses native EventSource API in browser.

4. IN-MEMORY EVENT BUS: Node.js EventEmitter used to decouple mutation
   handlers from SSE broadcasting. Simple and sufficient for single-process
   deployment.

5. OPTIMISTIC CONCURRENCY: Version field on work orders prevents lost updates
   when multiple users edit the same record. PATCH (not PUT) for safe partial
   updates – only provided fields are overwritten.

6. DARK GLASSMORPHISM THEME: Entire UI uses a dark zinc-950 base with
   translucent elements and backdrop blur. Designed for the shop floor
   environment with minimal eye strain.

7. QUERY STRING AUTH FOR SSE & EXPORTS: EventSource API doesn't support
   custom headers, so SSE authentication uses ?token= query parameter. Same
   approach used for export download links (<a href> can't set headers).

8. THREE-SHEET IMPORT/EXPORT: Excel files contain Schedule, Revisions, and
   Construction data in separate sheets. Import auto-detects sheet names and
   maps columns fuzzy-match style.

9. SINGLE TABLE FOR WORK ORDERS: v4.0.0 uses only engineering_work_orders
   (no separate general work_orders table). The "current_box" field serves
   as the assignee (maps from the Excel "Current Box" column).

10. ESM MODULES: The project uses ES modules throughout (type: "module" in
    package.json). Server files use import/export syntax.

19. KNOWN ISSUES & REMAINING WORK
================================================================================

1. NO ROLE-BASED ACCESS CONTROL: All authenticated users have equal access.
   There is no admin/viewer distinction.

2. SEARCH NOT FUNCTIONAL: The top bar search input in Layout.tsx is a
   placeholder – it doesn't filter or navigate anywhere yet.

3. NO PASSWORD CHANGE UI: Users cannot change their passwords through the
   app. Password changes require direct database modification.

4. SESSION CLEANUP: Expired sessions are only cleaned up when accessed.
   There is no background job to purge old sessions.

5. NO FILE ATTACHMENT SUPPORT: While the original design had an attachments
   concept, there is no attachment/file upload per work order.

6. CONSTRUCTION METRICS READ-ONLY: Construction metrics are imported via
   Excel but cannot be edited through the UI.

7. REVISION ALERTS READ-ONLY: Revision alerts are imported via Excel but
   cannot be created/edited/deleted through the UI.

8. ERP QUOTE URL YEAR: The erpQuoteUrl() function in utils.ts hardcodes
   "2025" in the URL path. This should be dynamic or configurable.
